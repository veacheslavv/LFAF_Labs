## Theory

### Parsing

**Parsing** is the process of analyzing a sequence of symbols according to the rules of a formal grammar. A parser takes the flat token stream produced by a lexer and reconstructs the hierarchical syntactic structure of the source text. Parsing is a fundamental stage in compilation, interpretation, and static analysis of programs.

The parsing process can be divided into two sub-tasks. **Syntactic analysis** verifies that the token sequence conforms to the grammar of the language. **Semantic analysis**, which often follows, checks whether the syntactically valid program actually makes sense — for example, that referenced variables are declared and that operand types are compatible. This laboratory focuses exclusively on syntactic analysis.

Formal grammars are commonly written in **Backus-Naur Form (BNF)** or **Extended BNF (EBNF)**. A grammar consists of a set of production rules, each mapping a non-terminal symbol to a sequence of terminals and non-terminals. The SQL subset parsed in this laboratory can be partially captured by rules such as:

```
program        ::= statement* EOF
statement      ::= select_stmt | insert_stmt | update_stmt
                 | delete_stmt | create_stmt | drop_stmt
select_stmt    ::= SELECT select_list FROM table_ref join_clause*
                   where_clause? group_by_clause? having_clause?
                   order_by_clause? limit_clause?
expression     ::= or_expr
or_expr        ::= and_expr ( OR and_expr )*
and_expr       ::= not_expr ( AND not_expr )*
not_expr       ::= NOT not_expr | comparison
comparison     ::= additive ( ( '=' | '<>' | '<' | '>' | '<=' | '>=' ) additive )*
additive       ::= multiplicative ( ( '+' | '-' ) multiplicative )*
multiplicative ::= unary ( ( '*' | '/' | '%' ) unary )*
unary          ::= '-' unary | primary
primary        ::= literal | identifier | function_call | '(' expression ')'
```

### Abstract Syntax Tree

An **Abstract Syntax Tree (AST)** is a rooted tree in which each interior node represents a syntactic construct — a statement, clause, or expression — and each leaf represents a terminal value such as a literal or identifier. The word "abstract" emphasizes that the tree encodes only semantically meaningful structure. Punctuation such as parentheses and semicolons is implied by the tree shape and is not stored explicitly.

The AST is distinguished from the **Concrete Syntax Tree (CST)**, also called a parse tree, which records every token including punctuation. A parse tree for `(a + b)` contains nodes for both parentheses, the `+` sign, and the identifiers. The corresponding AST contains only a `BinaryExpression` node with `+` as the operator and two `Identifier` leaves. The reduced representation makes downstream processing — such as type checking, optimization, and code generation — considerably simpler.

### Parsing Strategies

Practical parsers use one of two main strategies. **Top-down** recursive-descent parsers start from the root grammar rule and attempt to derive the input. They map directly to the grammar and are straightforward to implement by hand. Each non-terminal becomes a method; the body of the method reflects the right-hand side of the production rule. Top-down parsers can handle **LL(k)** grammars — grammars that can be parsed by reading at most *k* tokens ahead. Most programming language grammars, including the SQL subset here, are LL(1): a single look-ahead token is sufficient to decide which alternative to take.

**Bottom-up** parsers shift tokens onto a stack and reduce them to grammar symbols; they are usually generated by tools such as YACC or Bison. They can handle a broader class of **LR** grammars but are significantly harder to write by hand and produce less readable error messages. This laboratory implements a hand-written recursive-descent parser, which provides clear error messages and a direct correspondence between code and grammar rules.

### The Visitor Pattern

The **Visitor pattern** separates an algorithm from the object structure it operates on. In the context of an AST, it allows new operations — such as pretty-printing, type-checking, or SQL regeneration — to be added without modifying any node class. Each node class implements an `accept(visitor)` method that calls the corresponding method on the visitor. This is particularly valuable when the set of node types is stable but the set of operations is expected to grow.

### Lexer

`TokenType` is a Python `Enum` with one member per token category. Keywords such as `SELECT`, `WHERE`, and `JOIN` are listed individually. Operator tokens use short symbolic names: `EQ`, `NEQ`, `LT`, `GT`, `LTE`, `GTE`, `ADD`, `SUB`, `MUL`, `DIV`, `MOD`. Literals, identifiers, punctuation, and special tokens complete the set.

Every token category is matched by a compiled regular expression stored in an ordered list called `TOKEN_PATTERNS`. The lexer applies each pattern at the current position using `re.Pattern.match`, which anchors the match to that position. Multi-character operators such as `<=` and `<>` appear before single-character ones such as `<` to ensure correct priority. Similarly, `FLOAT_LITERAL` is placed before `INTEGER_LITERAL` so that `3.14` is not first consumed as the integer `3`.

After a match the lexer checks whether an `IDENTIFIER` token is actually a SQL keyword by looking up its uppercased value in a `KEYWORDS` dictionary. If found, the token type is replaced accordingly. Line and column numbers are tracked character by character so error messages can point to the exact source location.

```python
class TokenType(Enum):
    SELECT = auto(); FROM = auto(); WHERE = auto()
    AND    = auto(); OR   = auto(); NOT   = auto()
    EQ     = auto(); NEQ  = auto(); LT    = auto()
    GT     = auto(); LTE  = auto(); GTE   = auto()
    ADD    = auto(); SUB  = auto(); MUL   = auto()
    # ...

TOKEN_PATTERNS = [
    (TokenType.FLOAT_LITERAL,   re.compile(r'\d+\.\d+')),
    (TokenType.INTEGER_LITERAL, re.compile(r'\d+')),
    (TokenType.STRING_LITERAL,  re.compile(r"'(?:[^'\\]|\\.)*'")),
    (TokenType.NEQ,             re.compile(r'<>|!=')),
    (TokenType.LTE,             re.compile(r'<=')),
    (TokenType.EQ,              re.compile(r'=')),
    # ...
    (TokenType.IDENTIFIER,      re.compile(r'[A-Za-z_][A-Za-z0-9_]*')),
]
```

### AST Node Hierarchy

All nodes inherit from `ASTNode`, which provides an `accept(visitor)` method implementing the Visitor pattern. Each concrete node is a Python `dataclass`. The nodes are grouped by their role in the grammar.

**Literals** — `IntegerLiteral`, `FloatLiteral`, `StringLiteral`, `BooleanLiteral`, `NullLiteral` — carry a single typed value and are always leaf nodes. **References** — `Identifier`, `QualifiedIdentifier` for `table.column`, `Wildcard` for `*`, `Alias` for `expr AS name` — describe how columns and tables are named. **Expressions** — `BinaryExpression`, `UnaryExpression`, `BetweenExpression`, `InExpression`, `LikeExpression`, `IsNullExpression`, `FunctionCall` — represent computed values and predicates. **Clauses** — `SelectList`, `WhereClause`, `JoinClause`, `OrderByClause`, `GroupByClause`, `HavingClause`, `LimitClause`, `AssignmentClause` — represent named syntactic sections of a statement. **Statements** — `SelectStatement`, `InsertStatement`, `UpdateStatement`, `DeleteStatement`, `CreateTableStatement`, `DropTableStatement` — form the top-level nodes, and `Program` holds a list of statements as the root of the tree.

```python
@dataclass
class BinaryExpression(ASTNode):
    left: ASTNode
    operator: str
    right: ASTNode

@dataclass
class SelectStatement(ASTNode):
    select_list: SelectList
    from_table: Optional[TableReference]
    joins: List[JoinClause] = field(default_factory=list)
    where: Optional[WhereClause] = None
    group_by: Optional[GroupByClause] = None
    having: Optional[HavingClause] = None
    order_by: Optional[OrderByClause] = None
    limit: Optional[LimitClause] = None
```

### Parser

The `Parser` class consumes a `List[Token]` and exposes a single public method `parse()` that returns a `Program` node. Three low-level helpers manage the token stream: `check(*types)` tests the current token without consuming it, `match(*types)` consumes and returns the token only if it matches, and `expect(*types)` consumes or raises `ParseError` with a descriptive message including line and column.

Each grammar rule is a private method. The call graph mirrors the grammar directly:

```
parse → _statement → _select_stmt → _select_list → _expression
                   → _insert_stmt                → _or_expr
                   → _update_stmt                → _and_expr
                   → _delete_stmt                → _not_expr
                   → _create_stmt                → _comparison
                   → _drop_stmt                  → _additive
                                                 → _multiplicative
                                                 → _unary
                                                 → _primary
```

Operator precedence is encoded in the nesting depth. `OR` is at the top, then `AND`, `NOT`, comparison operators, additive, multiplicative, and finally unary and primary expressions. This is the standard precedence-climbing pattern for recursive-descent parsers.

Boolean and comparison operators are stored as symbols directly in the AST: `AND` becomes `&&`, `OR` becomes `||`, `NOT` becomes `!`, and `=` becomes `==`. This conversion happens in the parser at the point where each `BinaryExpression` or `UnaryExpression` node is constructed.

```python
def _or_expr(self) -> ASTNode:
    left = self._and_expr()
    while self.match(TokenType.OR):
        left = BinaryExpression(left=left, operator="||", right=self._and_expr())
    return left

def _and_expr(self) -> ASTNode:
    left = self._not_expr()
    while self.match(TokenType.AND):
        left = BinaryExpression(left=left, operator="&&", right=self._not_expr())
    return left
```

Special SQL predicates — `BETWEEN`, `IN`, `LIKE`, `IS NULL`, `IS NOT NULL` — are detected in `_comparison` after the left-hand operand has been parsed, and each is mapped to its own dedicated AST node type rather than being folded into a generic `BinaryExpression`.

### AST Printer

`ASTPrinter` is a concrete visitor that maintains an indentation level and builds a list of text lines. The helper `_child(label, node)` increases indentation, visits the node, then restores indentation. Since operator symbols are already stored in each node's `operator` field, the printer requires no conversion — it prints the field directly. The indented tree format closely mirrors the hierarchical structure of the AST and is easy to read without requiring any external libraries.

## Conclusions

The laboratory extended the Lab 3 SQL lexer with a `TokenType` enum and regex-driven tokenization, then built a complete recursive-descent parser on top of it. The parser converts flat token sequences into a hierarchical AST whose node types cover all six major SQL statement kinds together with the full expression language including arithmetic operators, boolean connectives, BETWEEN, LIKE, IN, IS NULL, function calls, and qualified identifiers.

Boolean and comparison operators are stored as programmer-readable symbols in the AST rather than SQL keywords, which makes the tree more intuitive to read and more convenient for downstream processing. The Visitor pattern used in `ASTPrinter` demonstrates a clean and extensible approach to tree traversal: adding a new operation such as a validator or code generator requires only a new visitor class without modifying any node definition. The recursive-descent structure made it straightforward to map grammar rules to code and to produce precise error messages containing line and column numbers.
