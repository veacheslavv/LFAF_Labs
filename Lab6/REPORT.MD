## Theory

**Parsing** is the process of analyzing a sequence of symbols according to the rules of a formal grammar. A parser takes the flat token stream produced by a lexer and reconstructs the hierarchical syntactic structure of the source text. It is a fundamental stage in compilation, interpretation, and static analysis, and the quality of a parser directly determines how well the rest of a language tool can reason about source code.

An **Abstract Syntax Tree (AST)** is a rooted tree in which each interior node represents a syntactic construct — a statement, clause, or expression — and each leaf represents a terminal value such as a literal or identifier. The word "abstract" emphasizes that the tree encodes only semantically meaningful structure. Punctuation such as parentheses, commas, and semicolons is implied by the tree shape rather than stored as explicit nodes. This makes the AST lighter and easier to traverse than a full concrete parse tree, which would preserve every token including whitespace and punctuation.

Practical parsers use one of two main strategies. **Top-down** recursive-descent parsers start from the root grammar rule and attempt to derive the input left to right. Each non-terminal in the grammar corresponds directly to a function in the parser, which makes the code straightforward to write, read, and debug. **Bottom-up** parsers such as LALR(1) work in the opposite direction: they shift tokens onto a stack and reduce them to grammar symbols when a complete right-hand side is recognized. Bottom-up parsers handle a larger class of grammars but are typically generated by tools such as YACC or Bison rather than written by hand.

This laboratory implements a hand-written recursive-descent parser. Recursive descent was chosen because the SQL subset being parsed fits comfortably within LL(1) with minor lookahead, the code maps directly to the grammar rules making it easy to verify correctness, and error messages can be made precise by tracking line and column numbers throughout the token stream. A generated parser would offer little benefit for a grammar of this size and would introduce an external dependency.

One important aspect of expression parsing in recursive-descent parsers is handling **operator precedence**. Without a mechanism to encode precedence, the parser would produce a flat or incorrectly associated tree for expressions like `a + b * c`. The standard solution is the **precedence climbing** technique, where each precedence level is represented by a dedicated parsing method that calls the method for the next tighter level when it needs an operand. Because tighter-binding methods are always called before looser-binding ones, the resulting tree naturally groups sub-expressions in the correct order.

The **Visitor pattern** is used to separate tree traversal from the node definitions themselves. Each node class implements an `accept(visitor)` method that calls the corresponding method on the visitor object. This makes it possible to add new operations over the AST — such as a type checker, an optimizer, or a code generator — without modifying the node classes. The alternative, placing all operations directly in the node classes, would cause the classes to grow indefinitely as new operations are added and would mix unrelated concerns in a single file.

### Lexer

`TokenType` is a Python `Enum` with one member per token category. SQL keywords such as `SELECT`, `FROM`, `WHERE`, `JOIN`, `GROUP`, `HAVING`, `ORDER`, and `LIMIT` are listed individually. Operator tokens use short symbolic names — `EQ`, `NEQ`, `LT`, `GT`, `LTE`, `GTE`, `ADD`, `SUB`, `MUL`, `DIV`, `MOD` — rather than verbose names like `EQUALS` or `GREATER_THAN`. Rare keywords that are not needed for the supported statement types, such as `OUTER`, `FULL`, `CROSS`, `REFERENCES`, and `FOREIGN`, were intentionally omitted to keep the token set lean. Data type names such as `INT`, `VARCHAR`, and `TEXT` are parsed as plain identifiers, which means the lexer accepts any type name without needing a dedicated token for each one.

Every token category is matched by a compiled regular expression stored in an ordered list called `TOKEN_PATTERNS`. The lexer applies each pattern at the current position using `re.Pattern.match`, which anchors the match to that position without scanning forward. Patterns are ordered by priority: multi-character operators such as `<=` and `<>` appear before their single-character prefixes `<` and `>` so that the longer match always wins.

After a successful match the lexer checks whether an `IDENTIFIER` token is actually a SQL keyword by looking up its uppercased value in a `KEYWORDS` dictionary. If found, the token type is replaced accordingly. The values `TRUE` and `FALSE` are handled separately as `BOOLEAN_LITERAL`. Line and column numbers are updated character by character inside each matched value so that every token carries an accurate source position for error reporting.

```python
class TokenType(Enum):
    SELECT = auto(); FROM = auto(); WHERE = auto()
    AND    = auto(); OR   = auto(); NOT   = auto()
    EQ     = auto(); NEQ  = auto(); LT    = auto()
    GT     = auto(); LTE  = auto(); GTE   = auto()
    ADD    = auto(); SUB  = auto(); MUL   = auto()
    DIV    = auto(); MOD  = auto()
    # ...

TOKEN_PATTERNS = [
    (TokenType.COMMENT,         re.compile(r'--[^\n]*|/\*.*?\*/', re.DOTALL)),
    (TokenType.FLOAT_LITERAL,   re.compile(r'\d+\.\d+')),
    (TokenType.INTEGER_LITERAL, re.compile(r'\d+')),
    (TokenType.STRING_LITERAL,  re.compile(r"'(?:[^'\\]|\\.)*'")),
    (TokenType.NEQ,             re.compile(r'<>|!=')),
    (TokenType.LTE,             re.compile(r'<=')),
    (TokenType.GTE,             re.compile(r'>=')),
    (TokenType.EQ,              re.compile(r'=')),
    (TokenType.LT,              re.compile(r'<')),
    (TokenType.GT,              re.compile(r'>')),
    # ...
    (TokenType.IDENTIFIER,      re.compile(r'[A-Za-z_][A-Za-z0-9_]*')),
]
```

### AST Node Hierarchy

All nodes inherit from `ASTNode`, which provides an `accept(visitor)` method that dispatches to the appropriate visitor method by constructing the method name from the node's class name. Each concrete node is a Python `dataclass`, which eliminates boilerplate and automatically provides `__repr__`. Optional fields default to `None` and list fields default to empty lists via `field(default_factory=list)` to avoid the shared-mutable-default pitfall.

The nodes are organized into six groups by their role in the grammar.

**Literals** — `IntegerLiteral`, `FloatLiteral`, `StringLiteral`, `BooleanLiteral`, `NullLiteral` — each carries a single Python value of the appropriate type.

**References** — `Identifier` for a bare column or variable name, `QualifiedIdentifier` for a dotted name such as `table.column`, `Wildcard` for `*` or `table.*`, and `Alias` for an expression paired with an `AS` name — describe how columns and tables are referenced in a query.

**Expressions** — `BinaryExpression` for two-operand operations, `UnaryExpression` for negation and unary minus, `BetweenExpression`, `InExpression`, `LikeExpression`, `IsNullExpression` for the SQL-specific predicate forms, and `FunctionCall` for aggregate and scalar functions — represent all computed values and conditions.

**Clauses** — `SelectList`, `WhereClause`, `JoinClause`, `OrderByClause`, `OrderByItem`, `GroupByClause`, `HavingClause`, `LimitClause`, `AssignmentClause` — represent named syntactic sections of a statement and group their child nodes under a descriptive label.

**Statements** — `SelectStatement`, `InsertStatement`, `UpdateStatement`, `DeleteStatement`, `CreateTableStatement`, `DropTableStatement` — are the top-level nodes that carry all clause children for one SQL command.

**Root** — `Program` holds a list of statements and is always the root of the tree returned by `parse()`.

```python
@dataclass
class BinaryExpression(ASTNode):
    left: ASTNode
    operator: str
    right: ASTNode

@dataclass
class SelectStatement(ASTNode):
    select_list: SelectList
    from_table: Optional[TableReference]
    joins: List[JoinClause] = field(default_factory=list)
    where: Optional[WhereClause] = None
    group_by: Optional[GroupByClause] = None
    having: Optional[HavingClause] = None
    order_by: Optional[OrderByClause] = None
    limit: Optional[LimitClause] = None
```

### Parser

The `Parser` class consumes a `List[Token]` and exposes a single public method `parse()` that returns a `Program` node. Three low-level helpers manage the token stream. `check(*types)` tests the current token type without consuming it. `match(*types)` consumes and returns the token only if its type is among those given, and returns `None` otherwise. `expect(*types)` consumes the token if it matches or raises a `ParseError` carrying a message that includes the expected types, the actual type, the source line, and the column number.

Each grammar rule is implemented as a private method whose name reflects the construct it parses. The call graph mirrors the grammar directly, so the code can be read alongside a BNF description of the language:

```
parse → _statement → _select_stmt → _select_list → _expression
                   → _insert_stmt                → _or_expr
                   → _update_stmt                → _and_expr
                   → _delete_stmt                → _not_expr
                   → _create_stmt                → _comparison
                   → _drop_stmt                  → _additive
                                                 → _multiplicative
                                                 → _unary
                                                 → _primary
```

Operator precedence is encoded directly in the nesting depth of method calls. `_or_expr` sits at the top and has the lowest precedence; `_primary` sits at the bottom and handles atoms. Each level calls the next level down for its operands, so a tighter-binding operator is always resolved before a looser one. This is the standard precedence-climbing pattern for recursive-descent parsers and requires no explicit precedence table.

Boolean and comparison operators are stored as programmer-readable symbols directly in the AST nodes rather than as SQL keyword strings. `AND` becomes `&&`, `OR` becomes `||`, `NOT` becomes `!`, and SQL `=` becomes `==` to distinguish equality testing from assignment. This conversion happens inside the parser at the moment each `BinaryExpression` or `UnaryExpression` is constructed, so no post-processing step is needed and the printer can print the `operator` field as-is.

```python
def _or_expr(self) -> ASTNode:
    left = self._and_expr()
    while self.match(TokenType.OR):
        left = BinaryExpression(left=left, operator="||", right=self._and_expr())
    return left

def _and_expr(self) -> ASTNode:
    left = self._not_expr()
    while self.match(TokenType.AND):
        left = BinaryExpression(left=left, operator="&&", right=self._not_expr())
    return left

def _not_expr(self) -> ASTNode:
    if self.match(TokenType.NOT):
        return UnaryExpression(operator="!", operand=self._not_expr())
    return self._comparison()
```

The `_primary` method handles the base cases of the expression grammar: all literal types, parenthesised sub-expressions, `NULL`, `TRUE`, `FALSE`, function calls recognized by a `(` following an identifier, qualified identifiers recognized by a `.` following an identifier, and wildcards. Function calls also support the `DISTINCT` modifier immediately after the opening parenthesis, which is needed to correctly parse expressions such as `COUNT(DISTINCT id)`.

Error handling is kept simple but informative. Every `ParseError` includes the token that caused the failure, its type, its source value, and its line and column number. This means that a malformed query such as one with a missing closing parenthesis or an unexpected keyword produces a message that points directly to the problematic token rather than to an arbitrary position in the stream.

The `_create_stmt` method deserves separate mention because column definitions are syntactically complex. After parsing the column name and type, the method enters a loop that consumes optional inline constraints in any order: `NOT NULL` sets the nullable flag to false, `PRIMARY KEY` is recognized by peeking at two consecutive identifiers with the right values, `UNIQUE` is a single identifier, and `DEFAULT` is followed by an arbitrary primary expression. This loop runs until it encounters a token that cannot start a constraint, at which point control returns to the column list parser which looks for a comma or closing parenthesis.

### AST Printer

`ASTPrinter` is a concrete visitor that maintains an integer indentation level and accumulates lines of text in a list. The `_child(label, node)` helper prints a label, increments the indent, visits the child node, then decrements the indent. The `_children(label, nodes)` helper does the same for a list. Because operator symbols are stored directly in the `operator` field of each expression node, the printer requires no translation table and simply formats the field inline with the node name, for example `BinaryExpr (&&)` or `UnaryExpr (!)`.

## Difficulties

The expression grammar required careful handling of the SQL-specific predicate forms. `BETWEEN low AND high` reuses the `AND` keyword as a separator rather than a boolean connective, so the parser must switch to a different sub-rule after recognizing `BETWEEN` to avoid consuming `AND` as a logical operator. Similarly, `NOT` can appear both as a standalone boolean operator and as a modifier for `BETWEEN`, `LIKE`, and `IN`. The solution was to consume an optional `NOT` token at the comparison level and then check which keyword follows it to determine which predicate form is being used. If none of the expected keywords appear after `NOT`, a `ParseError` is raised immediately.

Implicit table aliases — where `FROM users u` assigns the alias `u` without the `AS` keyword — required the parser to distinguish between an alias identifier and the start of the next clause keyword. This was solved by maintaining a set of clause-starting token types and checking that the next identifier is not one of them before treating it as an alias. Without this check the parser would incorrectly consume `WHERE` or `JOIN` as the alias of the preceding table.

String literals in SQL use single quotes and support escape sequences. The regular expression `'(?:[^'\\]|\\.)*'` handles both normal characters and backslash-escaped sequences within the string body. The outer quotes are stripped when constructing the `StringLiteral` node so the stored value contains only the content, not the delimiters.

Data types in `CREATE TABLE` statements such as `INT`, `VARCHAR(255)`, and `FLOAT` are recognized as identifiers rather than dedicated keyword tokens. This was a deliberate simplification: adding a token for every possible SQL type would bloat the token set without providing any benefit for the parser, since the type string is stored as-is in the `ColumnDefinition` node and not interpreted further. The only special handling is for the optional `(size)` suffix, which the column definition parser consumes explicitly when it detects an opening parenthesis after the type name.

## Conclusions

The laboratory extended the Lab 3 SQL lexer with a `TokenType` enum and regex-driven tokenization, then built a complete recursive-descent parser on top of it. The parser converts flat token sequences into a hierarchical AST whose node types cover all six major SQL statement kinds together with the full expression language including arithmetic operators, boolean connectives, BETWEEN, LIKE, IN, IS NULL, function calls, and qualified identifiers.

Storing boolean and comparison operators as programmer-readable symbols in the AST rather than SQL keywords makes the tree more intuitive to inspect and more convenient for any downstream processing stage. A future semantic analysis pass reading `&&` and `==` from `BinaryExpression` nodes requires no knowledge of SQL syntax, only of the symbol strings stored in the tree.

The Visitor pattern used in `ASTPrinter` provides a clean separation between node definitions and tree operations: adding a new pass such as a semantic validator or a query optimizer requires only a new visitor class and leaves all node definitions untouched. This is a significant advantage over an approach where each node class contains its own printing or evaluation logic, because it keeps unrelated concerns in separate files and makes the codebase easier to extend.

The recursive-descent structure kept the parser code readable and made it straightforward to produce error messages that include accurate line and column positions. The one-to-one mapping between grammar rules and parser methods means that when a rule needs to be adjusted — for example, to support a new clause or a new expression form — the change is localized to a single method rather than spread across a generated table or a set of action routines.
