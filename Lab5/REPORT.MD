## Theory

A **Context-Free Grammar (CFG)** is in **Chomsky Normal Form (CNF)** if every production rule has exactly one of the following forms:

- `A Ôćĺ BC` ÔÇö exactly two non-terminals on the right side
- `A Ôćĺ a` ÔÇö exactly one terminal on the right side
- `S Ôćĺ ╬Á` ÔÇö only if the start symbol can derive the empty string

CNF is a standard form used in formal language theory. It simplifies parsing algorithms such as CYK and is useful for proving properties of context-free languages. Every CFG can be converted to an equivalent CNF grammar through a series of well-defined transformation steps.

The normalization process consists of five sequential steps. Each step removes a specific class of problematic rules while preserving the language generated by the grammar.

## Implementation

### Overview

The program normalizes any context-free grammar to Chomsky Normal Form through five sequential transformation steps. Each step is implemented as a separate function that accepts a Grammar object and returns a new transformed Grammar object without mutating the input.

---

### Grammar Representation

The grammar is represented using a class with four fields: a set of non-terminals, a set of terminals, a dictionary of productions, and the start symbol.

```python
class Grammar:
    def __init__(self, VN, VT, P, S):
        self.VN = set(VN)
        self.VT = set(VT)
        self.P  = deepcopy(P)
        self.S  = S
```

Productions are stored as a dictionary mapping each non-terminal to a list of right-hand side strings. The empty string represents ╬Á.

---

### Step Functions

| Function | Description |
|---|---|
| `step1_eliminate_epsilon` | Removes all ╬Á-productions |
| `step2_eliminate_renamings` | Removes unit productions A Ôćĺ B |
| `step3_eliminate_inaccessible` | Removes symbols unreachable from S |
| `step4_eliminate_nonproductive` | Removes symbols that cannot derive terminals |
| `step5_to_cnf` | Converts remaining rules to binary CNF form |
| `to_cnf` | Runs all five steps in sequence |

## Step-by-Step Execution

### Step 1 ÔÇö Eliminate ╬Á-productions

First, all nullable symbols are identified. A symbol is nullable if it directly produces ╬Á or if all symbols in one of its rules are nullable.

Direct nullable: `B Ôćĺ ╬Á`, so **B is nullable**.

No further propagation applies since no other rule consists entirely of nullable symbols.

For every production containing B, a new version is added where B is omitted:

| Original rule | Added rule |
|---|---|
| `S Ôćĺ abAB` | `S Ôćĺ abA` |
| `A Ôćĺ BS` | `A Ôćĺ S` |
| `B Ôćĺ BA` | `B Ôćĺ A` |
| `B Ôćĺ ababB` | `B Ôćĺ abab` |

The rule `B Ôćĺ ╬Á` is removed. Result after Step 1:

```
S Ôćĺ abAB | abA
A Ôćĺ aSab | BS | S | aA | b
B Ôćĺ BA | A | ababB | abab | b
C Ôćĺ AS
```

---

### Step 2 ÔÇö Eliminate unit productions (renamings)

Unit productions are rules of the form `A Ôćĺ B` where a single non-terminal appears on the right side. For each symbol, its unit closure is computed ÔÇö the set of all non-terminals reachable through chains of unit rules. All non-unit rules from the closure are then added directly to the original symbol.

Unit rules found: `A Ôćĺ S`, `B Ôćĺ A`.

After resolving closures and removing unit rules:

```
S Ôćĺ abAB | abA
A Ôćĺ aSab | abAB | abA | BS | aA | b
B Ôćĺ aSab | abAB | abA | ababB | abab | BS | BA | aA | b
C Ôćĺ AS
```

---

### Step 3 ÔÇö Eliminate inaccessible symbols

Starting from the start symbol S, all reachable symbols are collected by traversing productions recursively.

Reachable from S: `S, A, B, a, b`

The symbols `C` and `D` are never referenced in any production reachable from S.

**Removed: {C, D}**

```
S Ôćĺ abAB | abA
A Ôćĺ aSab | abAB | abA | BS | aA | b
B Ôćĺ aSab | abAB | abA | ababB | abab | BS | BA | aA | b
```

---

### Step 4 ÔÇö Eliminate non-productive symbols

A symbol is productive if it can eventually derive a string consisting only of terminals.

All remaining symbols S, A, B can derive terminal strings through their existing rules. No symbols are removed in this step.

**Non-productive symbols: none**

---

### Step 5 ÔÇö Convert to CNF

Two sub-transformations are applied.

**5a. Replace terminals in long rules**

In any rule of length two or more, terminals are replaced by new non-terminals that produce them:

```
X1 Ôćĺ a
X2 Ôćĺ b
```

**5b. Break long rules into binary rules**

Any rule with more than two symbols on the right side is broken down into a chain of binary rules using helper non-terminals. For example:

```
A Ôćĺ aSab
```

After terminal substitution becomes `A Ôćĺ X1 S X1 X2`, which is then binarized:

```
X3  Ôćĺ X1 X2
X4  Ôćĺ S X3
A   Ôćĺ X1 X4
```

This process is applied to all rules until no rule has more than two symbols on the right side.

## Limitations

Multi-character non-terminal names such as X10 require a tokenizer that tries longer matches first. Naive character-by-character parsing would incorrectly split X10 into X, 1, 0. The number of helper non-terminals grows with the length of the original rules.

---

## Difficulties

Correctly handling the ╬Á-elimination step required generating all subsets of nullable positions within each rule. The tokenizer for multi-character non-terminal names needed careful implementation to avoid misparses during the binarization step. Ensuring no mutation of input grammars across steps required explicit deep copying.
